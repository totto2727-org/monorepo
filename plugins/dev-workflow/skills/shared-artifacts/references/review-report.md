# Reference: `review/<aspect>.md` の書き方

## 目的

**実装者と独立した視点**で、1 つのレビュー観点（`security` / `performance` / `readability` / `test-quality` / `api-design` / `holistic` の 6 観点）に特化して品質を検証する。`holistic` 観点は全体整合性チェック (Task Plan 完了判定 / `design.md` 整合性 / Intent Spec 成功基準充足見込み / 明白な bug の早期検出) を専任で担い、他観点は固有の深い検査を行う。

## 作成者 / 作成タイミング

- **作成者:** `reviewer` Specialist（観点ごとに別インスタンスで並列）
- **作成ステップ:** Step 7 (External Review)
- **承認:** ユーザー承認必須（Blocker 0 件の確認と Minor 扱い方針）

## ファイル位置

`docs/dev-workflow/<identifier>/review/<aspect>.md`

`<aspect>` ∈ {`security`, `performance`, `readability`, `test-quality`, `api-design`, `holistic`} (+ プロジェクト固有追加観点)

## 各セクションの書き方

### サマリ

| 深刻度  | 件数  |
| ------- | ----- |
| Blocker | {{n}} |
| Major   | {{n}} |
| Minor   | {{n}} |

**Gate 判定:** `approved` / `needs_fix` / `blocked`

### 指摘事項

各指摘に以下を付記:

- **深刻度:** Blocker / Major / Minor
- **該当箇所:** コミット SHA + ファイル + 行番号
- **問題の要約:** 何が問題か
- **根拠:** なぜ問題か（観点固有の理由）
- **推奨アクション:** 具体的な修正方針
- **設計との関連:** `design.md` の該当章節

### 深刻度の判定基準

| 深刻度      | 対応                                                                               |
| ----------- | ---------------------------------------------------------------------------------- |
| **Blocker** | Step 6 への差し戻し必須 (Task Plan 未実装 / Intent Spec 成功基準未達 / 明白な bug) |
| **Major**   | 修正方針 (Step 6 戻し / Retrospective 繰越) をユーザー判断で確定                   |
| **Minor**   | 記録のみで進行可 (Retrospective の材料として残す)                                  |

旧文書 (2026-04 以前のサイクルで生成された整合性レポート、現在は削除済み) で使われていた `High / Medium / Low` ラベルは廃止。再開する過渡期サイクルで旧ラベルを見かけた場合は、内容に応じて以下のマッピングで読み替える:

| 旧ラベル   | 新ラベルの目安                                          |
| ---------- | ------------------------------------------------------- |
| **High**   | 内容次第で `Blocker` (止め系) または `Major` (修正推奨) |
| **Medium** | `Major` または `Minor`                                  |
| **Low**    | `Minor`                                                 |

判定は機械的でなく内容ベース: Task Plan 未達 / 成功基準未達 / 機能不全は `Blocker`、修正余地ありは `Major`、改善提案は `Minor`。

### 修正ラウンド履歴

Step 6 ↔ Step 7 のループで Round 2 以降を行った場合は、Round 単位の Blocker / Major / Minor 件数推移と修正コミット SHA を記録する。3 周以上ループした場合は `dev-workflow/SKILL.md` の「ループ上限の目安」に従い、設計レベルの問題を疑って Step 3 ロールバックを Main / ユーザーに相談する判断材料となる。

| 項目                      | 記入内容                                                          |
| ------------------------- | ----------------------------------------------------------------- |
| `Round N`                 | Round 番号 (1 から始まる)                                         |
| `Blocker / Major / Minor` | Round 開始時点の指摘件数                                          |
| `主要指摘 (要約)`         | Round 内で最も影響度が高い指摘を 1〜2 行で要約                    |
| `修正コミット SHA`        | Round 内で Step 6 implementer が作成したコミット SHA (空白区切り) |

Round 1 のみで完了した場合は Round 1 行のみ記入し、Round 2 以降の行は省略可。`progress.yaml.review` のリストには各観点の最新 Round 集計のみ反映する (Round 履歴の真のソースは review/`<aspect>`.md 側)。

### 観点固有の評価項目

観点ごとに重点項目が異なる。該当観点の項目をテンプレートから引用して埋める。

#### security

- 認証・認可が全エンドポイントで適切に適用されているか
- 入力バリデーションが境界全て（HTTP, DB, 外部 API）で実施されているか
- 秘匿情報（秘密鍵、トークン、PII）がログや例外メッセージに漏れないか
- 依存ライブラリの既知脆弱性の確認

#### performance

- 計算量のオーダー評価（O(n²) 等の意図しない悪化がないか）
- N+1 クエリ等の I/O パターン問題
- メモリ使用量（大きなバッファ、メモリリーク候補）
- 並行性の正当性（データレース、デッドロック）

#### readability

- 名前が意図を表しているか
- 責務分離（SRP、関心の分離）
- コメントが「なぜ」を説明しているか（「何を」は避ける）
- 型が不変条件を表現しているか

#### test-quality

- エッジケース網羅（null、空、境界値、エラーパス）
- mock 使用が適切か（過剰な mock による prod 乖離リスク）
- テストの独立性（順序依存や共有状態の排除）

#### api-design

- 後方互換性（既存 API を破壊していないか、破壊する場合は代替経路と移行手順が揃っているか）
- 契約の明確さ（型や OpenAPI / JSONSchema 等でリクエスト・レスポンスが厳密に定義されているか）
- エラーモデル（例外 / エラーコード / エラーボディが一貫しているか、リカバリ方針が呼び出し側に伝わるか）
- 命名と拡張性（エンドポイント・フィールド命名規則、将来の追加を阻害しない設計か）
- ゲート契約（ユーザー承認 / Main 判定 / In-Progress 問い合わせの 3 分類が文書とテンプレートで一致しているか）

#### プロジェクト固有観点（該当する場合）

プロジェクトが独自のレビュー観点スキル（例: `effect-layer`, `totto2727-fp`）を提供している場合、本セクションで該当スキルの評価項目を引用する。dev-workflow の汎用観点と矛盾する場合は `dev-workflow` の「プロジェクト固有ルールとの関係」に従い**プロジェクト固有を優先**する。

### Scope フィールドの書き方

テンプレート冒頭の `**Scope:** {{review_scope}}` には、**担当観点のスコープ境界**を観測可能な形式で記述する。

- ✅ 「本サイクルで追加・変更された `src/auth/` 配下のコード。外部依存ライブラリの脆弱性は対象外」
- ✅ 「公開 API エンドポイント 5 件（`POST /auth/login` 等）のみ。内部ユーティリティは除外」
- ❌ 「セキュリティ観点全般」（スコープ境界が曖昧）
- ❌ 未記載／プレースホルダ残し

Main が Specialist 起動時に Scope 文言を明示しない場合、Specialist は作業開始前に Main に問い合わせる（`specialist-common` の入力契約に従う）。

### 他レビューとの整合性

他 reviewer と指摘が矛盾する場合はここに記録。Main が両者の根拠を比較して判断する材料となる。

**矛盾記録フォーマット:**

```markdown
- **矛盾対象:** `<other-aspect>.md` の指摘 #N
- **相手の主張:** <要約>
- **自分の立場:** <要約>
- **根拠の違い:** <技術的差異 / 優先度の違い / 観点境界の解釈違い のいずれか>
- **推奨:** <両者の指摘を個別に残す / どちらを優先すべきか Main 判断を依頼 / 観点を再分割>
```

## スコープ規律

- **1 Reviewer = 1 観点**。他観点には踏み込まない（`holistic` 観点は Round 2 以降のみクロスリファレンス参照可、Round 1 では独立並列）
- 実装の修正は reviewer の役割ではない（Main が implementer に差し戻す）

## 品質基準

| ✅ よい                                    | ❌ 悪い                                        |
| ------------------------------------------ | ---------------------------------------------- |
| 全指摘に具体的コミット SHA + 行番号がある  | 位置情報が曖昧                                 |
| 根拠が観点固有の技術的文脈で説明されている | 「改善の余地あり」のような無内容な指摘         |
| 推奨アクションがコードレベルで書かれている | 「検討せよ」のような投げっぱなし               |
| 他観点との境界を守っている                 | security の reviewer が performance 指摘を出す |

## 関連成果物

- **入力:** 全 Git コミット diff、`design.md` の関連部分、`intent-spec.md`
- **出力先:** Blocker があれば Step 6 へ差し戻し
- **並列性:** 観点ごとに独立。他 reviewer との直接通信は Main 経由
