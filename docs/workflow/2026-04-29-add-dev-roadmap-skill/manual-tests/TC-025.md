# TC-025: 仮想マイルストーン分解の説明性検証 (SC-11)

- **対象成功基準:** SC-11 (Intent Spec L114-117 — 任意の仮想ゴールを入力として 3 件以上のマイルストーン候補と各到達点・依存関係を追加情報なしに抽出できる説明性)
- **TC ID:** TC-025
- **実行主体 / 検証スタイル:** `manual` × `inspection`
- **実行タイミング:** Step 8 Validation (本サイクル)。`docs/workflow/2026-04-29-add-dev-roadmap-skill/validation-report.md` 作成時に Validator が実行する

## 目的

完成した dev-roadmap スキル本体および書き方ガイドを読んだ第三者 (Main / 別の Validation 担当者) が、**追加情報なし**で任意の仮想ゴールを 3 件以上のマイルストーンと依存関係に分解できることを目視判定する。本テストは「ドキュメントの説明性」そのものを検査するため自動化できず、人手による手順実行と再現性確認が必要となる。

## 前提条件

以下 3 ファイルが本サイクル完了時点でリポジトリに存在することを実行前に確認する。1 件でも欠落していれば本テストは実行不能のため Blocker として Main に報告する。

- `plugins/dev-workflow/skills/dev-roadmap/SKILL.md` (T1 成果物)
- `plugins/dev-workflow/skills/shared-artifacts/references/roadmap.md` (T7 成果物)
- `plugins/dev-workflow/skills/shared-artifacts/references/milestone.md` (T7 成果物)

確認コマンド例:

```bash
for f in \
  plugins/dev-workflow/skills/dev-roadmap/SKILL.md \
  plugins/dev-workflow/skills/shared-artifacts/references/roadmap.md \
  plugins/dev-workflow/skills/shared-artifacts/references/milestone.md; do
  test -f "$f" || { echo "MISSING: $f"; exit 1; }
done
```

## 入力

任意の仮想ゴール 1 件を Validator が選定する。再現性確保のため**最初に書き出した 1 行の文章**としてゴールを固定する。

- **推奨例:** `OAuth 認証導入 — 既存パスワード認証アプリに対し、Google / GitHub をプロバイダとする OAuth 2.1 認証を 1 リリースで導入する`
- **代替例:** 通知プラットフォーム導入、決済モダナイゼーション、マルチテナント基盤など、配下に複数の `dev-workflow` サイクルを必要とする規模のゴール

仮想ゴールの規模条件: **「単一サイクル (= 1 dev-workflow) で完結しないこと」が直感的に明らかであること**。単一サイクルで終わる粒度のゴールを選んだ場合は本テストの主旨を満たさないため別ゴールに差し替える。

## 手順

以下を**他の参照資料を一切開かずに**実施する (Web 検索・既存サイクル `task-plan.md` / `intent-spec.md` 等の参照を含めて禁止)。

### 手順 (a): 仮想ゴールの選択と固定 (上限 5 分)

1. 上記「入力」セクションから仮想ゴールを 1 つ選び、紙またはエディタに 1 行で書き出す。
2. 選定理由 (規模条件を満たす旨) を 1 行で添える。
3. 選定時刻を記録する (制限時間計測の起点)。

### 手順 (b): 検証対象 3 ファイルの通読 (上限 15 分)

1. 以下の順で 3 ファイルを通読する。読み返しは可。各ファイルにメモを残してよい (紙 / エディタ問わず)。
   1. `plugins/dev-workflow/skills/dev-roadmap/SKILL.md`
   2. `plugins/dev-workflow/skills/shared-artifacts/references/roadmap.md`
   3. `plugins/dev-workflow/skills/shared-artifacts/references/milestone.md`
2. 通読中に得られる「マイルストーン分解の指針」を以下の観点でメモすること:
   - マイルストーンの定義 (粒度・到達点の書き方)
   - 依存関係の表現方法 (DAG / Mermaid `graph LR`)
   - 統合検証マイルストーンの位置づけ
   - 仮想マイルストーン分解の説明性 (`references/roadmap.md` 該当セクションの 4 観点)
3. 15 分経過時点で通読を打ち切り、手順 (c) に進む。打ち切り後は 3 ファイルを再オープンしてよいが**ピンポイント参照のみ**とする。

### 手順 (c): マイルストーン候補の書き出し (上限 7 分)

1. 仮想ゴールに対し、**3 件以上**のマイルストーン候補を紙またはエディタに列挙する。各候補に以下を記述する:
   - **マイルストーン ID** (任意の短い英数ハイフン名、例: `m-oauth-provider-spike`)
   - **タイトル** (1 行)
   - **定性的到達点** (1〜3 行、観測可能な観点で書く必要はない。戦略層レベルで「何が達成された状態か」を書く)
2. ゴールの全範囲をカバーするように候補を選ぶ (1 候補のみで全体を覆ってはならない)。
3. 統合検証マイルストーン (最終マイルストーン) を含むことを推奨する (`references/milestone.md` 配置パターン例参照)。

### 手順 (d): 依存関係の決定 (上限 3 分)

1. 手順 (c) のマイルストーン候補間の依存関係を矢印 (Mermaid または手書きの矢印図) で描く。
2. グラフが DAG であることを確認する (循環があれば手順 (c) に戻る)。
3. 起点候補 (依存先がない / 最も少ない) と最終マイルストーン候補 (依存元がない) が 1 つ以上ずつ存在することを確認する。

### 手順 (e): 制限時間内に完了したことの確認

1. 手順 (a) の選定時刻から手順 (d) 完了までの**経過時間が 30 分以内**であることを記録する。
2. 30 分を超えた場合は超過時間と原因 (例: 「マイルストーン依存判定に時間を要した」) をメモする。**30 分超過は FAIL 判定の主要因の 1 つ**となる。

## PASS 判定基準

以下の **5 条件すべて** を満たすときのみ PASS と判定する。

1. **マイルストーン件数 ≥ 3:** 手順 (c) で書き出された候補が 3 件以上である。
2. **各マイルストーンに定性的到達点が記述されている:** タイトルだけのエントリは 0 件 (1 行以上の到達点記述があること)。
3. **依存関係が明示されている:** 手順 (d) のグラフが描かれており、ノードと矢印で表現された有向グラフである。
4. **DAG (有向非巡回グラフ) が成立している:** 循環依存がない。
5. **制限時間 30 分以内に完了している:** 手順 (a)〜(d) を 30 分以内に完了した。

加えて、PASS と判定された結果に対し、**Main または別の Validation 担当者** が同じ手順 (a)〜(e) を独立に実施した結果と「実質的に同等のマイルストーンセット」が導出されることをレビュアー 1 名が確認する (qa-design.md TC-025 row の判定基準)。

「実質的に同等」の定義:

- **個数:** 双方とも 3 件以上
- **粒度:** マイルストーンのカバー領域 (例: 「プロバイダ統合」「セッション管理」「移行計画」) が一致する (個別文言の完全一致は不要)
- **依存方向:** 起点候補→最終マイルストーンの DAG 構造が一致 (枝の本数や中間ノードのトポロジーが類似)

レビュアーが「実質的に同等でない」と判断した場合は FAIL とする。

## FAIL 判定例

以下のいずれかに該当する場合は FAIL とする。FAIL 時はその根拠と発見事項を `validation-report.md` に記録し、対象ファイル (T1 / T7 成果物) の改善要否を Main に報告する。

- **件数不足:** マイルストーン候補が 2 件以下しか書き出せない (PASS 条件 1 違反)。
- **到達点未記述:** タイトルのみで定性的到達点が空のマイルストーンが 1 件以上ある (PASS 条件 2 違反)。
- **依存関係未記述:** 矢印 / グラフが描かれていない、またはマイルストーンが羅列のみで関係性が表現されていない (PASS 条件 3 違反)。
- **循環依存:** DAG ではない (PASS 条件 4 違反)。
- **制限時間超過:** 手順 (a)〜(d) で 30 分を超過した (PASS 条件 5 違反)。
- **ファイル不参照では分解不能:** 手順 (b) の通読を途中で中断し、別資料 (Web 検索・既存サイクル成果物等) を参照しないと分解できなかった (= 説明性不十分のシグナル)。
- **再現性欠如:** Main または別の Validation 担当者が独立に実施した結果と「実質的に同等」でないとレビュアーが判定した。
- **戦略層に観測可能基準が混入:** 定性的到達点の代わりに「N 件以上の OAuth 認可成功」等の観測可能な数値基準を書いてしまった場合は、`references/roadmap.md` の品質基準違反であり、ドキュメント側の説明不足を示唆する FAIL シグナル。

## 結果記録テンプレート

Validator は以下の項目を `validation-report.md` 内の対応セクションに直接記録する (本サイクルでは補助ディレクトリ `validation-evidence/` は使わずインライン化方針):

- 仮想ゴール (1 行)
- 開始時刻 / 終了時刻 / 経過時間
- マイルストーン候補リスト (ID + タイトル + 到達点)
- 依存グラフ (テキスト Mermaid または図のスクリーンショット)
- PASS / FAIL 判定 + 根拠
- 独立レビュアーの判定結果 (PASS の場合のみ)
