This document is the **essential test design finalized in Step 4**. During Step 6 (Implementation) the implementer appends "tests discovered during implementation" as `TC-IMPL-NNN`. See `share-artifacts/references/qa-design.md` for authoring details.

This cycle is a **documentation / plugin restructuring** cycle (no production code paths added). Therefore "tests" here mean **observable acceptance checks** executed by general-purpose CLI commands (`bash`, `find`, `wc`, `grep`, `jq`, `deno`, `vp`), not by a test framework runner (Vitest / Playwright / pytest are out of scope for this cycle).

## Overview

The Intent Spec defines ten observable success criteria (SC-1 through SC-10) covering: (a) old-path deletion, (b) new-path presence, (c) the 300-line hard cap on the two new entry SKILL.md files, (d) 3-tier relative-link integrity, (e) consistency of the three marketplace.json variants, (f) absence of leftover hardcoded references to old skill / plugin names, (g) `deno check` exit 0 on the two Deno scripts, (h) consistency of the three `plugin.json` variants, (i) `.claude/settings.json` `enabledPlugins` updates, and (j) `vp check` exit 0.

Each SC is restated below in observable terms and assigned an `SC-N` ID.

- **SC-1 (old paths gone):** `find` reports zero results for each of `plugins/moonbit/`, `plugins/components-build/`, `.agents/skills/effect-layer/`, `.agents/skills/effect-runtime/`, `.agents/skills/effect-hono/`, `.agents/skills/totto2727-fp/`.
- **SC-2 (new paths present):** all expected files under `plugins/totto2727-coding/` exist (manifest, four SKILL.md files, seven `coding/references/` files, three `test/references/` files, two scripts, two slash commands).
- **SC-3 (300-line cap):** `wc -l plugins/totto2727-coding/skills/coding/SKILL.md` and `wc -l plugins/totto2727-coding/skills/test/SKILL.md` each return a value `<=` 300.
- **SC-4 (3-tier link integrity):** every Markdown relative link found in `coding/SKILL.md`, `test/SKILL.md`, `coding/references/<lang>-skill.md`, `test/references/<lang>-skill.md` resolves to an existing file. Per design A10, the **External skill references** section (everything from the heading `## External skill references` onwards) is **excluded** from this scan because it intentionally lists external skills by name only, with no Markdown link.
- **SC-5 (3 marketplace.json variants consistent):** `.claude-plugin/marketplace.json` (edit base), `.cursor-plugin/marketplace.json` (sync derivative), and `.agents/plugins/marketplace.json` (Codex sync derivative) all contain the `totto2727-coding` entry and contain neither `moonbit` nor `components-build` entries; each in its respective schema.
- **SC-6 (no hardcoded references):** repository-wide `grep` finds zero remaining references to old plugin paths or old skill names (`plugins/moonbit/`, `plugins/components-build/`, `.agents/skills/effect-`, `.agents/skills/totto2727-fp`, `effect-layer`, `effect-runtime`, `effect-hono`, `totto2727-fp`, `moonbit-bestpractice`, `moonbit-docs`, `components-build-docs`) outside the immutable corpus (`docs/adr/**`, `docs/workflow/**`, `node_modules/**`, `.git/**`).
- **SC-7 (`deno check` exit 0):** `deno check plugins/totto2727-coding/.script/generate-docs-moonbit.ts` and `deno check plugins/totto2727-coding/.script/generate-docs-components-build.ts` both exit with code 0.
- **SC-8 (3 plugin.json variants identical):** `.claude-plugin/plugin.json`, `.codex-plugin/plugin.json`, `.cursor-plugin/plugin.json` under `plugins/totto2727-coding/` have identical `name` / `description` / `version` / `author` fields after `c-plugin dev marketplace sync`.
- **SC-9 (`enabledPlugins` updated):** `.claude/settings.json` `enabledPlugins` contains `"totto2727-coding@totto2727": true`, has neither `"moonbit@totto2727"` nor `"components-build@totto2727"`, and retains `"dev-workflow@totto2727": true` and `"totto2727@totto2727": true`.
- **SC-10 (`vp check` exit 0):** `vp check` from repo root exits with code 0 (formatting / lint / type check pass).

A separate qualitative criterion — **plugin actually discoverable by Claude Code in a fresh session** — is recorded as a `manual × scenario` smoke test (TC-022) because (i) it cannot be automated within the worktree (per `research/plugin-discovery-mechanism.md` Implications #4: directory-source plugins resolve against the main checkout, not the worktree), and (ii) it is the ultimate user-facing acceptance check for the consolidation. It does not itself satisfy any SC (the SC-2 file-presence check is the formal proof) but is recorded so the validator (Step 8) can run it after the PR merges to `main`.

## Rationale for automated vs. manual

The vast majority of checks are **`automated × assertion`** because each SC reduces to either "a file/path exists or does not exist", "a numeric threshold (line count) holds", "a JSON document parses and contains an expected key", "a `grep` returns no matches", or "a CLI command exits 0". All of these are reproducible from a single shell invocation with no human judgment, fully suitable for CI.

A small set is **`automated × scenario`** because the verification involves a sequence of operations (e.g. SC-5 requires (i) running `c-plugin dev marketplace sync`, then (ii) parsing three JSON files and (iii) cross-checking their contents — although the running of the sync itself is a Step 6 implementation task; the QA check verifies the post-sync state).

One check is **`manual × scenario`** (TC-022, plugin discovery smoke test): the cycle is being developed in a worktree, but Claude Code reads `marketplace.json` from the main checkout (`research/plugin-discovery-mechanism.md` Findings #3 + Implications #4). Automating "open a fresh Claude Code session and check that `totto2727-coding:coding` appears in the available skills list" inside the worktree is not possible without merging the PR first; this is a post-merge sanity check best run by a human (or AI agent) opening a session in the main checkout.

We adopt **no `ai-driven` tests** for this cycle: every check is either purely deterministic (CLI exit codes / file presence / JSON contents) or requires a real Claude Code session start (which is `manual` in our environment).

We adopt **no `automated × inspection` combinations** (this combination is forbidden anyway). We adopt no `automated × observation` either, because no SC requires threshold judgment on metrics — SC-3's "line count <= 300" is a discrete `assertion` (string equality of `wc -l`'s integer output against a numeric upper bound), not metric observation.

## Test file placement policy

This cycle adds **no test framework files** (no `*.test.ts`, no `e2e/`). Acceptance checks are executed as inline shell commands or, where reusable, as small ad-hoc shell snippets that the validator (Step 8) records in `validation-report.md` along with their outputs.

- **`automated × assertion` checks (TC-001 through TC-019):** executed inline by the Step 8 validator using the commands listed in each TC's `Pass criterion` column. No persisted test files.
- **`automated × scenario` checks (TC-020, TC-021):** also executed inline, but record the multi-step command transcript in `validation-report.md` to make the sequence auditable.
- **`manual × scenario` smoke check (TC-022):** the procedure document is recorded under `docs/workflow/2026-05-04-totto2727-coding-plugin/manual-tests/TC-022-plugin-discovery.md` (created by Step 6 implementer or Step 8 validator on first execution).

This placement policy honors the cycle's nature (documentation-only, no runtime code surface). If a future cycle adds runtime code, the standard project-wide `vp test` placement (per `vite-plus` skill) takes over, but it does not apply here.

### Enum value quick reference

- **Actor (`Actor` column):** `automated` | `ai-driven` | `manual`
- **Style (`Style` column):** `assertion` | `scenario` | `observation` | `inspection`
- **Forbidden combination:** `automated × inspection` (not allowed) — none used in this design.
- **Combinations requiring notes (△):** `ai-driven × assertion`, `manual × assertion`, `manual × observation` — none used in this design.

The single △-adjacent combination used here is `manual × scenario` (TC-022). Per the reference taxonomy `manual × scenario` is a recommended (✓) combination for "manual scenarios" and does not require a Notes-column rationale, but a rationale is provided anyway because the worktree-vs-main-checkout caveat is load-bearing.
