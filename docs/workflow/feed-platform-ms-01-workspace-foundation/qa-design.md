## Overview

ms-01 は **実装本体を持たない雛形整備サイクル**であり、Intent Spec の SC-1〜SC-11 (intent-spec.md L131-152) は次の 6 カテゴリに分類される:

1. **構造的検証** — ディレクトリ / ファイル配置 / `package.json` / `tsconfig.json` / `wrangler.jsonc` 等の存在 (SC-1, SC-5, SC-9 中心)
2. **コマンド成功検証** — `vp run check` / `vp run test` / `vp run -r build` の exit code + 期待出力 (SC-2, SC-3, SC-4)
3. **コード内含検証** — `effect-layer` / `effect-runtime` / `ManagedRuntime` パターンの使用、Logger.unwrap + Env.Service 経由の Logger 形式判定、`await using` パターン採用 (SC-6 + design 確定 refinement)
4. **artefact 起票検証** — ADR-01 / ADR-02 のファイル存在 + 主要セクション充足 (SC-7, SC-8)
5. **CI 検証** — GitHub Actions の `vp run --parallel ci` PASS (SC-10)
6. **ロードマップ進捗連携検証** — `roadmap-progress.yaml` の `ms-01-workspace-foundation` を `completed` に遷移可能な前提充足 (SC-11)

各 SC の観測可能形は intent-spec.md に既記載のため再記述しないが、design.md の確定事項により以下の 3 件は **追加観測対象** として TC に展開する:

- **3 プロジェクト共通 Effect skeleton 5 ファイル** (CC-7, design.md L930-960): `feature/env.ts` (Layer.succeed) + `feature/greeting.ts` (Layer.sync) + `feature/health.ts` (Layer.effect) + `feature/runtime/server.ts` (ManagedRuntime + Layer 合成) + `feature/runtime/hono.ts` (await using middleware)
- **Logger 形式選択方法** (Alternatives Option P, design.md L1022): `Layer.unwrap` + `Env.Service` 経由で実行時判定し、`import.meta.env.PROD` の直参照は **不採用**。Env Service の値ソースは **`process.env.NODE_ENV`** (wrangler / vite 自動設定) を採用し、`wrangler.jsonc.vars.ENV` の独自 vars は **不採用** (Step 7 → Step 8 user-gate refinement で再確定、本番 deploy で dev 値焼き込みバグ予防)
- **Runtime 解放パターン** (Alternatives Option S, design.md L1025): TC39 `await using` 自動破棄を採用し、try/finally + 明示的 `runtime.dispose()` 呼び出しは **不採用**
- **Web UI レンダリング戦略** (design.md L1044, L1062-1069): ms-01 では素朴な `c.render(<Document>...)` のみで、`createPageOrFrame` パターンは **未採用** (ms-04 / ms-07 委譲)

SC-11 は SC-1〜SC-10 全充足の論理的帰結であり独立した観測対象を持たないため、TC では「SC-1〜SC-10 全 PASS の前提下で `roadmap-progress.yaml` の milestone エントリが `completed` 化可能な状態 (= blocker フィールドが空 / 必要な ADR 参照リンクが存在)」を検証する。

## Rationale for automated vs. manual

ms-01 の検証対象は **ほぼ全てが構造的 / コマンド成功 / 静的内容パターンマッチ**であり、人間の主観的判断 (UX / 視覚的一貫性) を要する観測点は存在しない。よって既定方針は **`automated × assertion` 一択**。例外を以下にまとめる。

**`automated × scenario` (Effect Service test)**: SC-3 で各プロジェクトに配置する `feature/<name>.test.ts` (Effect Service の `Health.check` / `Greeting.greet` を Layer 合成 + `runPromise` で実行) は **複数 Layer の合成 → Effect 実行 → 結果照合** という連続ステップを伴うため `scenario` に分類する。Vitest 等の test runner で完全自動再現可能なため `automated`。

**`automated × observation` (`find` カウント / 出力内含チェック)**: SC-5 (`find ... -name 'worker.ts' | wc -l` ≥ 2) と SC-6 (各プロジェクト内に `Layer` / `ServiceMap.Service` / `ManagedRuntime` を使う TS ファイル ≥ 1) は **数値閾値 / パターンマッチ件数の閾値判定**を行うため `observation` に分類する (assertion より範囲指定が含まれる観測点)。

**`automated × inspection` は禁止組合せ**のため採用しない。ADR-01 / ADR-02 の本文質的妥当性 (= 文章として説得力があるか) は本サイクルの検証範囲外 (Operational rules: 「ADR の中身は検証対象外、ファイル存在 + 主要セクション充足のみ」)。主要セクション (Status / Context / Decision / Consequences / References) の **見出し存在チェック** に限定し、これは grep / awk による pattern match で `automated × assertion` として扱える。

**`manual` / `ai-driven` は採用しない**: 個人開発 + 雛形整備サイクルという性格上、人間判断を要する観測点を残すと CI / Step 8 validator の自動検証経路が不完全になる。すべての TC は `vp run`、`find`、`grep`、`gh run view` 等のコマンドで再現可能な observable に落とす。

## Test file placement policy

ms-01 では以下の **配置方針のみ** を確定する。具体ファイルパスは Step 5 (Task Decomposition) / Step 6 (Implementer) の責務。

- **`automated × scenario` (Effect Service test)**: 各プロジェクトの `feature/` 配下に **検証対象 module ごとに `<name>.test.ts` を colocation 配置** (`vite-plus/test` 経由、Vitest 設定ファイルは新設しない、CC-5)。Step 7 → Step 8 user-gate refinement で旧設計のソース直下 `smoke.test.ts` 1 ファイル形から、検証対象 module の隣に置く形に変更
  - `js/app/feed-platform-backend/src/feature/health.test.ts` (Health 1 件、describe `'Health'`、relative import `./env.ts` `./health.ts`)
  - `js/app/feed-platform-web/app/feature/greeting.test.ts` (Greeting 1 件、describe `'Greeting'`、relative import `./greeting.ts`)
  - `js/app/feed-platform-web/app/feature/health.test.ts` (Health 1 件、describe `'Health'`、relative import `./env.ts` `./health.ts`)
  - `js/app/identity-provider/app/feature/greeting.test.ts` / `js/app/identity-provider/app/feature/health.test.ts` (web と同形)
  - 合計 5 件 (backend 1 + web 2 + IdP 2)
- **`automated × assertion` / `automated × observation` (構造検証 / コマンド成功検証 / 内含検証)**: Step 8 の validator が **コマンド実行ベース**で再現する。コードとしてのテストファイルは作成しない (= validator の `validation-evidence/` に実行ログとして残す)
  - 配置先 (= 観測ログ): `docs/workflow/feed-platform-ms-01-workspace-foundation/validation-evidence/<TC-ID>.log` (Step 8 で生成、Step 4 では空)
- **CI 検証 (SC-10)**: GitHub Actions の `vp run --parallel ci` ジョブ結果を `share-ci-monitoring` 二重チェックで観測。テストコード自体は新設不要 (既存 CI ワークフロー流用、CC-9)

### TC で網羅していない design 確定事項 (意図的)

以下の design 確定事項は **観測対象として独立 TC を立てない** 判断:

- **CC-1 catalog 利用** (`catalog:effect` 等): TC-003 (`vp run check` 通過) に内包 (catalog 解決失敗時は typecheck エラー)
- **CC-2 `imports.#@/*` 規約**: TC-003 / TC-004 に内包 (誤った import path は typecheck / `feature/<name>.test.ts` で発覚)
- **CC-3 tsconfig 共通形** (`extends: ['@totto2727/fp/tsconfig/vite']`): TC-003 に内包 (strictest 適用が外れると typecheck エラー範囲が変動)
- **CC-4 Lint / Format 集約**: TC-003 に内包 (Ultracite ルール違反は exit ≠ 0)
- **wrangler.jsonc 共通設定** (compatibility_date / observability / placement): TC-008 / TC-020 構造検証に内包 (ファイル存在確認の延長で、内容検証はフォローアップサイクルに委譲)
- **Effect skeleton の specific Layer kind 配分** (succeed / sync / effect): TC-012 (5 ファイル存在) で構造的に担保。Layer kind 区別の TC を分離するとファイル粒度依存になり、後続マイルストーンの軽微な変更で偽陽性となるため不採用

### Enum value quick reference

- **Actor (`Actor` 列):** `automated` のみ採用 (`ai-driven` / `manual` は本サイクル不採用)
- **Style (`Style` 列):** `assertion` (中心) / `scenario` (TC-004 のみ) / `observation` (TC-006 / TC-008 / TC-011 の 3 件)
- **禁止組合せ:** `automated × inspection` (採用なし)
- **△ 組合せ:** 採用なし (`Notes` の rationale 記載は不要)
